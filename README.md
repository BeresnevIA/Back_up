## Задание 1. 

Кейс

Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.

Необходимо описать, какие варианты резервного копирования подходят в случаях:

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

1.1. Восстановление за предыдущий день (RPO = 24 часа)

Здесь допустима потеря данных за текущий день. Логика: нужна одна цельная точка восстановления на вчерашний вечер.

Дифференциальный бэкап — раз в неделю полный, ежедневно — разница с полным. Для восстановления нужны два файла (полный + вчерашний диф). Быстрее, чем ежедневный полный.

Полный бэкап ежедневно — проще логически (один файл = одна копия), но дольше делается и больше места ест.

Снапшоты раз в сутки — моментальные снимки системы, восстановление — откат к снимку. Быстро, но зависит от платформы (VM, SAN, облако).

Ключевое различие: при диф. бэкапе восстановление чуть сложнее (два файла вместо одного), но окно бэкапа короче.

1.2. Восстановление за час до поломки (RPO = 1 час)

Потеря максимум часа данных. Требует частых операций и обычно — комбинации подходов.

Варианты:

Инкрементный бэкап с шагом 1 час — редкий полный, затем каждый час — только изменения с предыдущего бэкапа (любого). Экономит место, но восстановление медленное: нужно склеить цепочку из 24+ файлов, и если один битый — всё после него не восстановить.

Журнал транзакций (архивирование логов каждый час) — после полной копии регулярно бэкапим только лог изменений. Восстановление точное до минуты, но нужно разбираться в режимах восстановления БД (FULL recovery model).

Частые снапшоты (каждый час) — самый простой в восстановлении (выбрал снимок за нужный час и откатился), но самый жадный до места. И не все БД «любят» снапшоты на дисковом уровне — может потребоваться согласование.

Гибрид: полный раз в день + инкремент каждый час + сброс логов. Стандарт для многих финансовых систем.
